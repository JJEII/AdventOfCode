** Day  3: Lobby – Maximize joltages from battery banks
Part 1

Find the maximum individual battery joltage within the bank of batteries.

If it's the last battery ...

... find the maximum joltage preceding it, and combine them for the total.

If not ...

... find the maximum joltage succeeding it, and combine them for the total.

Part 2

While solving Part 1, it occurred to me that this is exactly the direction Part 2 would go. You need to be a bit more clever for this one.

An initial important observation is that ...

... the first digit of the final result must be the maximum possible, while also leaving enough batteries afterward to fill out all remaining required digits.

Extending this further leads to the observation that ...

... this is true for every digit being merged into the final result.

This is, of course, in conjunction with the fact that ...

... each maximized digit must also follow all preceding already-maximized digits.

** Day  4: Printing Department – Count forklift-reachable rolls of paper
Part 1

It's fairly straightforward to load the data into a grid; run through each location in it; check that it's a roll and, if so, assess what's in its immediate surroundings in order to determine if it's reachable or not; counting the reachable results the whole time, as you go. Although, that may be a little exacting in terms of getting the details right, it's not "hard" to do.

A better way ...

Instead of loading everything into a grid, if you load into a set only the (row, col) coordinates of just the rolls of paper, then you will subsequently iterate only through rolls (skipping all empty locations), and you only need to check if immediately surrounding (row, col) locations are in that set when counting how many rolls are adjacent to each.

Part 2

For the most part, this is the same as Part 1, just repeated, and with some adjustments on each iteration. Until nothing changes any more.

In the case of the "better" way in Part 1 ...

You just remove every reachable roll in each iteration to set up for the following iteration, until there's nothing more to remove, counting along the way (or just remembering the original size, then subtracting the final size from it at the end).

Part 2

This one's a bit more tricky, but not too bad.

Fundamentally, the problem to solve is ...

... counting every "fresh ID" exactly once, instead of multiple times, because the provided ranges overlap with each other in various ways, and they must be processed as ranges because simply iterating through all IDs between the minimum range low value and the maximum range high value isn't feasible due to how large the range values are.

There are two obvious ways to resolve overlapping ranges ...

... shrink ranges, as needed, when they overlap, so they no longer do. But, this doesn't necessarily work very well in some situations, such as if some ranges exist entirely inside other ranges, or at least are entirely overlapped by some combination of other ranges. And, even if you do get this working properly, it still will generally yield more resulting ranges than when using the next approach.

... merge ranges, as needed, when they overlap, so they no longer do. But, then the challenge is to identify subsets of all ranges that are merge-able into respectively representative single ranges.

There are several ways to approach the merge problem, but probably the best is ...

... to first ascendingly sort all the ranges by their low values (with tie-breaks by their end values). See sorted(...), or [].sort().

Then, iterate through them, tracking ...

... an intermittently changing start value and whether or not the currently assigned end value is ≥ to the following range's low value. (Technically, you can merge on end ≥ low-1, but you don't need to.)

If it is, then set end to ...

... max(end, high), and continue.

If it's not, then add (start, end) to a set of merged ranges, and set start and end to the current range's low and high, respectively, to begin merging a new range, then continue.

Once done, simply sum the lengths of all merged ranges to get the total count of fresh IDs.

Note that it is also possible to accumulate totals on the fly (which should be more performant than storing merged ranges and later summing them), but beware of accounting for the final sorted-input range because its factoring-in with regard to merging or not, while inside versus outside the loop, is dependent upon the input data.

I had initially misread the problem to be asking for the maximum area of all "in bounds" rectangles, so I had written a "solution" to enforce that no data-points were inside a valid rectangle, no segments intruded into/across any valid rectangle, and no valid rectangle possessed any edge overlapping by more than one unit with a boundary segment with opposite winding direction. This just happened to turn up the correct answer for the problem's input data, because that data did not well-exercise the problem's constraints.

But, this is not, in fact, what the problem asks for: "Using two red tiles as opposite corners, what is the largest area of any rectangle you can make using only red and green tiles?"

This formulation allows the boundary to enter/exit/wind-around-inside a max rectangle so long as there is never any 'gap' between the intruding boundary lines!

With this very different way of understanding the problem and thinking about how to solve it (also quite differently), I wrestled for a while with how on Earth to try to tackle it.

I eventually landed on the idea of ...

... subdividing any selected rectangle into "empty regions" within it (i.e., regions that are left over inside the rectangle after carving it up by "removing" overall-boundary edges), and then testing each of those regions for "inside-ness" versus "outside-ness". If any of them ends up as "outside", the entire rectangle is invalid; if all are "inside", the rectangle is valid.

To test "inside-ness", I ...

... performed a horizontal ray-cast out of each region to beyond the dataset's limits, counting boundary/edge crossings along the way. An odd count means "inside", an even count means "outside".

Note that doing this comes with its own complications due to the possibility of the ray aligning exactly along some number of horizontal edges, in which case it matters what the edges connecting to each end are doing.

However, this is fairly straightforwardly resolved with the trick of ...

... offsetting the ray's y-coordinate by a small amount to avoid alignment issues—for example, by ±0.5.

Note: When I eventually developed this solution, it ran more slowly than I would have liked (a few minutes?). I know there are some things I could do to speed it up, but also it may be better to precompute (for the entire dataset) all the "outside" regions first (via almost exactly the same technique, but on a bounding-box rectangle for the whole dataset), then during the main run, test each selected rectangle for intersections with any of those "outside rectangles". If there's an intersection found, the selected rectangle is invalid; if not, it's valid. Maybe this would be better, and maybe not. (The "outside" rectangles could, for example, be arranged in a tree structure for efficient querying, or whatever... but, I haven't tested this idea.)
It was not clear to me how exactly to carve up rectangles, but the way I ended up doing it was ...

... by performing a "sweep" from left to right across the rectangle, evaluating the state of a vertical cross-section only when it changes—so, at every vertical segment (sorted by x), and in-between them if there's a gap.

For each unit-wide "vertical slice", I ...

... already knew which vertical segments were in-play (since I'd previously done the bookkeeping for that), so I sliced through the horizontal segments to bring in all of those also in-play for that x-value slice (all clipped to the selected rectangle's bounds, of course). I represented all the vertical and horizontal segments as (low, high) y-ranges (horizontal-edge "y-ranges" being only size 1), y-sorted them all, and performed a "range merge" much like what was done for Day 5 Part 2 (also merging ranges directly juxtaposed). The result was the ranges of 'filled' locations, which I used to produce an inverted set of ranges for what was empty within that slice of rectangle (if anything).

For each of these slices' empty ranges, I could have tested "inside-ness" using the same ray-cast technique described above. However, I didn't want to produce legions of these, with large subsets of them likely parts of the same "empty regions", and then test every single one of them for "inside-ness" rather than just testing every "empty region" as a whole, just once each.

To accomplish this single ray-cast per region, I...

... used union-find to merge adjacent (with successive slices) empty ranges into larger "empty regions", then performed a single ray-cast test only on each region's "representative slice-range".

Big performance-improvement spoilers related to my note above...

In order to (drastically!) speed things up, I implemented a couple of important changes, with all the heavy lifting being done as precomputation.

The first was to create "compressed coordinate" mappings for both x and y directions of the input data, where all the unique x and y values mapped to their densest possible versions while still accurately representing the input structurally. Basically, I started each at 0 and incremented it by 1 if the difference between successive sorted values was only 1, and by 2 if it was ≥2.

The second was to build summed-area table from this compressed-coordinate representation of the input. I initialized the whole table with zeros then ran my inside/outside region code on the entire compressed-coordinate dataset in order to identify "outside" regions, which I then filled into the table with 1 values throughout. From this intermediate table, I constructed a final summed-area table.

To assess each selected rectangle during the main run, I simply performed an O(1) lookup in the summed-area table, using the compressed-coordinate versions of the rectangle's coordianates. If the result is nonzero, some part of the rectangle is "outside"; if it's zero, then it's fully inside.

I kept the reverse-sorting-by-area code structure in place, too, so the first area found that works, is the largest area.

These changes dropped the runtime down to almost nothing (perhaps a couple seconds on my rather old laptop).

A suggestion for finding all maximal solutions is ...

... to precompute all the data-point-pair areas, making a dictionary to map every distinct area to all the pairs that produce it, then iterate through the areas in reverse-sorted-area order (then all the pairs for each), so that the very first valid area found is the answer (and all such answers for that area).

The single most important thing to recognize is ...

... that this is fundamentally a linear programming/algebra problem.

The first step is to ...

** Day 12: Christmas Tree Farm – Trying to fit collections of oddly-shaped gifts under limited-boundary trees
Part 1

Carefully analyze the input data. (Seriously. Just do it. That's the first hint.)

As far as I know, this problem is ...

... in the general case, NP-hard. So, when I glanced at the scale of the provided input data, I was initially entirely at a loss as to how to approach it because the values appear far too large to just brute-force through.

However, I figured that, at the very least, I could examine the data in more detail to see if maybe some of the inputs could fairly easily be ruled out via a simple check or two.

The first test was to ...

... see if any of the entries were able to be rejected even if they were perfectly densely packed (no holes at all).

For each data input independently examined, let:

  • W : width of boundary region
  • L : length of boundary region
  • AR = WL : area of boundary region
  • ni : required number of shape i
  • ai : area covered by shape i, computed as its '#'-count (same across all "WxL: ni for i in 0..5" data inputs)
  • AD = Σ ni ai : "dense area", which is the theoretical minimum coverage area of all of an input's shapes (ignoring region shape)
  • N = Σ ni : total shape count
It should be obvious that if AD>AR, then that input can immediately be rejected as impossible.

Imagine my surprise when literally hundreds of the inputs were rejected by this simple check!

Frustratingly, when the problem first became available to work on, I had only tried this test on the very first input, and by hand, and I'd accidentally reversed the comparison and concluded that the boundary region area was just slightly larger than the total shape-coverage area, even though it was actually slightly smaller! Oof!

At any rate, this was all fine and good to discover, but it still didn't necessarily get me much closer to a final answer since it didn't tell me whether or not those remaining (not rejected) would work, and trying to crank through even just one of them would be computationally infeasible.

But, it did encourage me to then look more closely at those remaining inputs. I next examined ...

... just how much larger AR was than AD. It turned out to be by quite a substantial amount, even for the smallest one! Now, this seemed potentially promising...

I decided to examine them in terms of ratio, too—(1 - AD/AR)—and this also showed great promise because even the smallest "excess" was still >26%! That felt like quite a lot of "wiggle room".

Given all this, I decided to just see what would happen if ...
